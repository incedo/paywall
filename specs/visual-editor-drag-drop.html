Visual Wall Editor — Block Composition & Drag-and-Drop

Follow-up spec · builds on ADM-11/12/17 · prefix VWE-* · 2026-06-14

This spec turns ADM-11's "visually composes wall designs from a constrained block library" into a concrete block model, a drag-and-drop editor, and the missing-parts backlog. It is written after a code review of main (16 BCs, 523 tests, 96% coverage): the structured-output and no-JS guarantees of ADM-11 are met, but the composition interaction ADM-11 describes is not yet built.

Scope boundary (unchanged). The editor produces a structured wall definition (JSON), never code or arbitrary HTML/JS (GAP-10, BP-01). Drag-and-drop changes only how a business user arranges constrained blocks; it does not widen what can be rendered. The gate component still renders server-side and the no-JS guarantee (BP-01) is preserved.

1. Where we are today

The current implementation models a wall as a flat, fixed-slot template and edits it with a form, not a block canvas.

Honest note on the design package. The original Wall Designer.dc.html comp was itself a config-panel + live-preview form, not a drag-and-drop canvas. Drag-and-drop is an intention stated in the requirement (ADM-11 "visually composes … from a constrained block library"), not something the design exploration prototyped. This spec is therefore additive design work, and §6 flags the product decisions that need sign-off before build.

2. The block model (VWE-01..05)

Replace the fixed-slot template with an ordered list of typed blocks. The flat WallConfig maps deterministically onto a default block list, so existing walls keep working (migration, VWE-05).

// shared/commonMain — pure Kotlin, serializable, multiplatform
@Serializable
sealed interface WallBlock { val id: String }

@Serializable data class Headline(override val id:String, val text:String): WallBlock
@Serializable data class BodyCopy(override val id:String, val text:String): WallBlock
@Serializable data class PriceDisplay(override val id:String, val planRef:String? = null): WallBlock // price filled by CEP at render
@Serializable data class CtaButton(override val id:String, val label:String, val role:CtaRole): WallBlock
@Serializable data class LoginLink(override val id:String, val label:String = "Log in"): WallBlock
@Serializable data class ImageBlock(override val id:String, val url:String, val alt:String = ""): WallBlock
@Serializable data class LegalText(override val id:String, val text:String): WallBlock
@Serializable data class ConsentBlock(override val id:String): WallBlock            // AC-14
@Serializable data class MeterIndicator(override val id:String): WallBlock          // PW-22/23
@Serializable data class SocialProof(override val id:String, val text:String): WallBlock

enum class CtaRole { PRIMARY, SECONDARY }

@Serializable
data class WallLayout(val blocks: List<WallBlock>)   // ordered; the editor's output
IDPriRequirement
VWE-01MUST A wall design holds an ordered WallLayout of typed blocks drawn from the constrained library above. No free-form block type may be added without a spec change (this is the "constrained" in constrained block library).
VWE-02MUST Structural validity is enforced server-side on save (ADM-01 — the console owns no logic): a gate layout MUST contain at least one value-proposition block (Headline or BodyCopy) and at least one PRIMARY CtaButton (PW-03); ConsentBlock and PriceDisplay are singletons; ConsentBlock, if present, MUST be first (AC-14 "distinct step"); LegalText MUST follow the CTAs.
VWE-03MUST Block edits are events on the wall stream (extends ADM-13): WallLayoutChanged carries the new ordered layout; versioning/rollback/audit apply unchanged. New events registered in paywallSerializersModule.
VWE-04SHOULD Per-block translatable text (ADM-15): each text-bearing block carries a locale map; the renderer picks the brand locale, falling back to default. Extends the existing WallCopy override to block granularity.
VWE-05MUST Back-compat migration: an existing flat WallConfig deterministically projects to a default WallLayout (Headline=title, BodyCopy=body, PriceDisplay, CtaButton(primary), CtaButton(secondary)/LoginLink, optional ImageBlock/LegalText/ConsentBlock). Old walls render identically; round-trip test guards it.

3. The drag-and-drop editor (VWE-10..18)

A three-zone workspace: palette (left, the constrained library), canvas (centre, the live gate the blocks compose), inspector (right, props of the selected block). This reuses the existing CrmTheme tokens and the live-preview machinery already in WallDesignerScreen.

IDPriRequirement
VWE-10MUST Palette → canvas drag: dragging a block type from the palette and dropping it on the canvas inserts a block at the drop index. Compose detectDragGestures in commonMain so it works on web (Wasm), desktop (JVM) and touch targets (multiplatform rule).
VWE-11MUST Reorder within the canvas: a placed block can be dragged to a new position; the layout order updates live and the preview recomposes.
VWE-12MUST Keyboard / non-pointer fallback (NFR-13, WCAG 2.1 AA): every drag action has an equivalent button/keyboard path — select block, "move up / move down / remove". Drag-and-drop is an enhancement, never the only way to arrange a layout (also covers environments where pointer-drag is awkward).
VWE-13MUST Constraint feedback during composition: invalid drops are rejected with an inline reason (VWE-02), e.g. a second ConsentBlock or a LegalText dropped above the CTAs. The publish button is disabled while the layout is structurally invalid.
VWE-14MUST Per-block inspector: selecting a block shows only its typed props (Headline → text; CtaButton → label + role; ImageBlock → url + alt). Replaces the flat content form.
VWE-15SHOULD Multi-context live preview (ADM-12): the canvas previews the composed layout per breakpoint (mobile/desktop), per brand theme (ADM-10), and per visitor state (anonymous/registered/subscriber) — the gate contexts of ADM-12.
VWE-16SHOULD In-editor accessibility lint (ADM-17): contrast (text vs brand background), tap-target size, missing image alt, and heading order are flagged on the offending block before publish; publish is blocked on AA failures.
VWE-17COULD Templates from layouts (ADM-16): a composed layout can be saved as a template and instantiated for another brand, keeping brand theme tokens separate from block layout/copy.
VWE-18COULD Undo/redo of composition actions within an editing session (the event log already gives cross-session history via ADM-13; this is in-session ergonomics).

4. Multiplatform & rendering notes

  • One source tree. The drag layer uses Compose pointer/gesture APIs available in commonMain, so web (Wasm) and desktop (JVM) share it; mobile/touch inherits it later. No platform-specific editor branch (multiplatform rule).
  • Renderer stays declarative. The gate component renders a WallLayout by mapping each block to a CrmTheme-tokened composable — the same renderer the live preview uses and the same one the public gate uses. One renderer, no drift, no-JS preserved (BP-01).
  • Server is authoritative. Structural validity (VWE-02) and accessibility gating (VWE-16) are re-checked on the save/publish endpoint; the editor's checks are UX, not the security boundary (ADM-01).

5. Build order (suggested slices)

  1. Model + migration (VWE-01/05): WallBlock/WallLayout, default projection from flat config, round-trip + projection tests. No UI yet — old editor keeps working.
  2. Block renderer: one composable per block type; gate preview + public gate both render from WallLayout. Visual parity test against the current fixed layout.
  3. Inspector + add/remove/reorder by buttons (VWE-12/14): the keyboard-accessible path first — this alone delivers block composition without drag.
  4. Drag-and-drop layer (VWE-10/11/13): palette drag + canvas reorder on top of the button path.
  5. Multi-context preview + a11y lint (VWE-15/16), then templates/undo (VWE-17/18).

Slice 3 is the pragmatic MVP: it satisfies ADM-11's "constrained block library" composition (add/remove/reorder, structured output) and WCAG before any drag code exists; drag-and-drop (slice 4) is then a pure enhancement.

6. Open product decisions (need sign-off)

QuestionWhy it mattersRecommended default
Is full free-arrangement wanted, or a fixed skeleton with optional blocks? Free arrangement is more work and more ways to make an ugly/illegal wall; a constrained skeleton (value prop → price → CTAs → legal) matches PW-03 and most publisher walls. Constrained skeleton with optional/insertable blocks and enforced ordering (VWE-02), not a blank canvas.
Drag-and-drop now, or button-based reorder first? The accessible button path (VWE-12) delivers the requirement; drag is ergonomics. Ship slice 3 (buttons) first; drag in the next slice.
Per-block locale (VWE-04) vs per-wall locale (current WallCopy)? Block-level is finer but heavier; the experiment is nl-NL only today (ADM-15). Keep per-wall WallCopy for the experiment; design the block model so per-block can be added without a schema break.
