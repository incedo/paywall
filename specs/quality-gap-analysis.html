Quality & Requirements Gap Analysis

Review of main · prefix Q-* · 2026-06-14

Headline: the codebase is in good shape. 16 bounded contexts, 523 tests (0 failures), 96.2% backend line coverage, stable-only versions enforced, and the security/robustness basics already present — rate limiting (AC-06, RequestRateLimiter), webhook signature verification (WebhookVerifier), idempotent webhooks (tested), and staff RBAC + step-up on ~20 admin endpoints. The findings below are refinements, not alarms. Each was verified against a real file or build output; nothing here is speculative.

1. Build & environment robustness

IDSevFinding & suggested improvement
Q-1HIGH Build breaks under a non-UTF-8 locale. BDD test methods use back-tick names with em-dashes (e.g. `Scenario - Grace period — past_due still gets access (SUB-05)`). The Kotlin compiler writes a .class file whose name contains —; on a container whose sun.jnu.encoding defaults to ANSI_X3.4-1968 (ASCII, no LANG set) this throws InvalidPathException and the whole build fails. Verified: the gate failed on the default locale and passed only with LANG=C.UTF-8.
Fix: pin a UTF-8 locale in the build/CI image, add -Dfile.encoding=UTF-8 -Dsun.jnu.encoding=UTF-8 to org.gradle.jvmargs, or keep test method names ASCII-only. The encoding flag is the most portable.
Q-2LOW Configuration cache off. Gradle suggests enabling it; the build is ~1.5 min warm. Low priority, pure speed.

2. Coverage transparency

IDSevFinding & suggested improvement
Q-3MED The 96% excludes the production event-store adapter. backend/build.gradle.kts excludes persistence/** from the JaCoCo report — that is PostgresEventStore, the real DB adapter (append-condition / advisory-lock / partition logic). Its integration tests are gated on PAYWALL_TEST_PG_URL and don't feed the report, so the headline number reflects everything except the hardest infra. Not wrong, but the number reads more reassuring than it is.
Fix: run the Postgres integration tests in CI (a PG service container) and include persistence/** in the report, or report it as a separate, explicitly-tracked figure so the exclusion is visible rather than implicit.
Q-4LOW Other exclusions are defensible but broad. Typed-ID value classes, generated serialization (EventSerializationKt), the JWKS-fetching Companion and KratosCiamSessionClient are excluded — all reasonable (value classes / JaCoCo-unreachable / live-infra). Keep them, but add a one-line comment per exclusion stating why so the list stays auditable as it grows.
Q-5LOW Line coverage ≠ behaviour coverage. Several projection rate getters (VariantStatsProjection, CohortProjection, offer acceptance) guard divide-by-zero but are mostly happy-path tested. Add a boundary test per rate (zero denominator, single sample) where not already present.

3. Security & robustness (already strong — spot refinements)

IDSevFinding & suggested improvement
Q-6OK Verified present: origin trust (INF-02) intercepts all but /health; staff RBAC + AAL2 step-up across admin endpoints; webhook signature verification on /integration/cep-advice; rate limiting (AC-06); idempotent ad-completion and duplicate webhooks (tested). No action — recorded so the next reviewer doesn't re-flag these.
Q-7MED Confirm signature verification is uniform across all inbound webhooks. cep-advice verifies the signature; audit that every machine-to-machine inbound (entitlements ingest, ad-completion, account-deletion, offer callbacks) either verifies a signature or is provably covered by origin trust + a secret. List them in one place so coverage is obvious; add a test that an unsigned/forged body is rejected for each.
Q-8LOW Idempotency is per-handler, not systemic. Dedup currently relies on decision models ignoring duplicates (ad-completion) or DCB. Consider a shared idempotency-key check on integration endpoints (store the provider event id; reject replays) so a new webhook can't silently double-count.

4. Requirements correctness (verified divergences)

These are not polish — each is a behaviour that diverges from a MUST requirement, confirmed by reading the cited line on main.

IDSevRequirement & verified divergence
Q-12HIGH MT-04 / MT-05 — bots and prefetch reads still burn a meter credit. In AccessService.kt the isBot flag (line 140) flows only into logWallEvent as a log label (used at lines 404/426/443); the metering write at :286-288 fires on decision.countsTowardMeter regardless of isBot. Only isVerifiedCrawler (line 175) short-circuits. MT-04 says a read counts "not for … bots, or prefetch requests"; there is also no prefetch (Sec-Purpose) detection at all.
Fix: when isBot (and not a verified crawler) or prefetch, serve teaser/gate without counting — mirror the verified-crawler branch. Add a regression test per strategy.
Q-13HIGH FGA-01 / FGA-02 — a null-expiry grant lives forever. GrantDecision.kt:47: it.expiresAtEpochMs == null || it.expiresAtEpochMs > now — a null expiry is treated as a perpetual live grant, and GrantIssued.expiresAtEpochMs defaults to null. FGA-02 requires every grant be time-bound (expires_at required). The admin path defaults to 30 days, but the type and decision model permit/honour a permanent grant (and three non-admin issuance paths append GrantIssued directly).
Fix: make expiresAtEpochMs non-nullable and fail closed on a missing expiry; route share-token/ad/data-gate issuance through one issueGrant() that applies the FGA-02 TTL ceiling and FGA-03 per-source cap.
Q-14MED NFR-20 — ES256 tokens are silently rejected. CiamJwtValidator.kt:91,93 casts the JWK to RSAPublicKey and builds only Algorithm.RSA256; an EC key returns unknown_key → AC-07 anonymous. NFR-20's allowlist is "RS256/ES256". If Ory Hydra is configured for ES256, every genuine subscriber silently loses entitlement.
Fix: branch on JWK key type — ECDSA256 for ECPublicKey, RSA256 for RSA; keep none/HS* rejected.
Q-15MED MT-06 — rolling_30d meter period is a silent no-op. AccessService.kt:490-491: both the rolling_30d and else arms return currentPeriod() (calendar month). An operator configuring a rolling window gets calendar-month behaviour with no error.
Fix: implement the rolling window (needs the period/tag scheme the code comment notes) or reject rolling_30d at config-load (API-03) so the config cannot silently lie.
Q-16MED NFR-20 — audience check is opt-in and off by default. The aud validation runs only when CIAM_AUDIENCE is set; an unconfigured deployment accepts tokens minted for any other Hydra client. NFR-20 makes aud a MUST.
Fix: fail startup if a JWKS issuer is configured without an audience, rather than silently disabling a MUST check.
Q-17MED NFR-11 / FGA-05 — the decide path does not fail open. The event-store queries in /api/v1/decide and /articles/{id} have no try/catch; a store/Keto outage propagates as a 500 instead of degrading (FGA-05: proceed without grants; NFR-11: fail open). INF-06 places fail-open at the edge Worker (not in this repo), so this is partly an architecture-boundary call, but the service-level guarantee is currently unmet origin-side.
Fix (if pursued): wrap the decide path to log the incident and return teaser/gate for premium (full for free) on infrastructure failure.
Q-18MED DG-03 — data-gate consent withdrawal is unmodelled. Only DataGateConsentGiven exists; there is no withdrawal event/endpoint and nothing consults withdrawal before issuing future data-gate grants, so "withdrawal stops future use" cannot be met (the TTL-bound non-retroactive half is fine).
Fix: add DataGateConsentWithdrawn + endpoint and reject new grants for a withdrawn subject+purpose.

6. Overall assessment

This is a mature, broadly-tested codebase — access-decision core, metering, entitlements, FGA, analytics, payments, partners, brands, offers and the admin surface are all present, event-sourced, with the security basics (RBAC, step-up, rate-limit, webhook signatures, idempotency) in place. But "523 tests / 96%" should not be read as "no correctness gaps": the review found a handful of real MUST divergences, of which two are high-impact — Q-12 (bots/prefetch burn meter credit, a direct MT-04/MT-05 violation that also skews AN-* funnel data) and Q-13 (a null-expiry grant is permanent, against FGA-02). Q-14 (ES256 rejected) is a latent outage waiting on a CIAM config choice. These are narrow, well-localised fixes — but they show that high line-coverage masked behaviours the tests simply never asserted. On the build side, Q-1 (ASCII-locale fragility) should be fixed before it bites a fresh CI runner. The largest product gap is the unbuilt drag-and-drop composer (Q-9), specified separately. Recommended order: Q-1, then Q-12/Q-13/Q-14, then the remaining MED items, then the visual-editor build.
