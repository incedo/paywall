Implementation Plan — Quality Fixes & Visual Editor

Derived from the two review specs · 2026-06-14 · sizes are rough (½d / 1d / 2–3d / 1wk)

This plan sequences the work found in Quality & Requirements Gap Analysis (Q-*) and Visual Wall Editor — Block Composition & Drag-and-Drop (VWE-*) into four phases. Order is by risk reduced per unit effort: stop the build fragility and the two real correctness bugs first; then the remaining MUST divergences; then build the visual composer (accessible MVP before drag-and-drop). Each row names its verifying test so "done" is unambiguous.

Working agreement. One phase per feature branch, ~7–10 small commits, full suite green before each commit (LANG=C.UTF-8 ./gradlew :shared:jvmTest :backend:test + a local PG 16), cite the requirement ID in every commit. PRs opened per completable slice. No alpha/beta deps (verifyStableVersions). New domain events registered in paywallSerializersModule.

Sequencing & dependencies

  • P0 → P1 are independent of the editor and can ship immediately; P0 first (build + the two real bugs). Together ≈ 1 week.
  • P2 depends on nothing in P0/P1 but is the long pole; its first slice (VWE-01/05 model + projection) unblocks everything else and is safe (no behaviour change). P2 ≈ 2 weeks.
  • P3 depends on P2 (drag needs the block model + accessible reorder underneath). P3 ≈ 2–3 weeks.
  • Net: correctness/robustness closed in ~1 week; a requirement-complete visual composer (P2) in ~3 weeks; full drag-and-drop experience (P3) in ~5–6 weeks elapsed, parallelisable across two people (P0/P1 + P2 concurrently).

Open product decisions (block before P2/P3)

DecisionRecommended default
