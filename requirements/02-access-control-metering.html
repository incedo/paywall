Doc 2 — Access Control & Metering

This is the core engine of the experiment: the component that decides, for every article request, whether the visitor gets the full article or the gate.

1. Access decision flow

request(article, visitor)
  → article.content_tier == free?            → FULL CONTENT
  → visitor has valid entitlement?           → FULL CONTENT      (AC-01..03, EA-*)
  → partner access (IP allowlist/federation)?→ FULL CONTENT      (PA-*, IPW-*)
  → FGA grant for (subject, article)?        → FULL CONTENT      (FGA-01..06, AG-*, DG-*)
  → variant = experiment assignment          (EX-*)
  → decision = paywall_strategy[variant](visitor, article)
      hard      → GATE
      freemium  → GATE
      metered   → meter not exhausted? count & FULL : GATE
      dynamic   → score ≥ threshold? GATE : FULL (with floor rule PW-42)
  → log wall event (AN-*)

2. Enforcement (AC)

IDPriorityRequirement
AC-01MUSTThe access decision is made server-side. The full premium article body is never present in the HTML/JSON delivered to an unentitled visitor. Hiding content with CSS/JS only is not acceptable.
AC-02MUSTEntitlements are checked on every premium content request against the subscription store (see SUB-*); an expired or canceled subscription loses access at the end of its paid period.
AC-03MUSTEntitlement check results may be cached per session for at most 5 minutes to limit latency, but cancellation/expiry must take effect within that window.
AC-04MUSTPremium API endpoints (e.g. article JSON) enforce the same rules as HTML pages — no unauthenticated side door.
AC-05SHOULDThe teaser fragment (PW-02) is generated server-side from the article body, so the cut-off point cannot be manipulated by the client.
AC-06SHOULDRate-limit and log suspicious access patterns (e.g. one identity used from many IPs concurrently) for later account-sharing analysis.
AC-07MUSTLogged-in identity is established exclusively from a CIAM-issued JWT validated per NFR-20..24. A request with a missing, invalid, or expired token is treated as anonymous (falls back to visitor_id metering) — never as an error page.
AC-08MUSTThe JWT authenticates who the user is; what they may access (entitlements) is resolved from the local subscription store (AC-02). Entitlement claims inside the token, if present, are advisory only — cancellations must take effect within the AC-03 window regardless of token lifetime.

3.1 Progressive registration (PR) — CIAM

IDPriorityRequirement
PR-01MUSTIdentity data is collected in steps, stored as Kratos identity traits: step 1 = email + password only (US-05); later steps (name, interests, newsletter opt-in) are separate Kratos flows triggered at configured moments — by the paywall (e.g. after Nth read) or by the CEP (trigger context profile_enrichment).
PR-02MUSTThe registration wall (PW-50) uses step 1 only. Every later step states its purpose and is skippable unless it is the explicit price of a data-gated grant (DG-*).
PR-03SHOULDProfile completeness is exposed (claim or profile API) so the CEP can segment on it; each collected trait carries a consent/purpose record (AN-20).

3.2 Adaptive authentication (AA) — CIAM

IDPriorityRequirement
AA-01MUSTSensitive operations — payment method change, email/password change, account deletion, user-scoped grant administration — require a step-up authentication level (AAL2: TOTP/WebAuthn/email code via Kratos), asserted by the acr/amr claims and enforced origin-side in the shared middleware (NFR-22).
AA-02SHOULDRisk signals (new device, impossible travel, Cloudflare bot score via INF-09) can force re-authentication at login. Signal evaluation lives in the CIAM login flow, not in paywall code.
AA-03MUSTFriction guard: reading content never requires step-up — AAL1 (password session) is always sufficient for article access. Adaptive auth protects the account, not the content.

3.3 Entitlement authorisation (EA) — origin, with JWT pattern

Recommended JWT pattern: the JWT proves identity; entitlements are data. But to avoid a database hit on every request, entitlement claims may ride along in the access token as a cache:

IDPriorityRequirement
EA-01MUSTThe subscription store remains the single authority for entitlements (AC-02/AC-08). Any entitlement representation elsewhere (token claims, edge cache) is derived and disposable.
EA-02SHOULDA Hydra token hook embeds entitlement claims (e.g. ent: ["basic"], tier) into access tokens at issuance/refresh. If these claims are used in access decisions, access-token TTL must be ≤ the AC-03 freshness window (5 min) — otherwise claims are display-only.
EA-03SHOULDThe edge Worker may use valid entitlement claims for the fast path (e.g. routing subscribers past the cached gated rendering); the origin re-derives entitlements for the actual full-content response. Edge trusts claims for routing, origin trusts only the store for serving.
EA-04MUSTSubscription changes (upgrade UP-14, cancellation, pause SUB-07) revoke/refresh outstanding tokens where the provider supports it, and in all cases take effect within the AC-03 window via token expiry.

3.4 Partner authorisation (PA) — FGA + CIAM federation

IDPriorityRequirement
PA-01MUSTA partner (company, school, library, telco bundle) is an FGA object holding a partner subscription: Keto tuples model user —member→ partner and partner —subscriber→ tier; the access check resolves transitively, so individual users need no personal subscription.
PA-02MUSTPartner users join via (a) CIAM federation — the partner's IdP added as OIDC/SAML provider in Kratos, membership tuple written on first federated login — or (b) IP allowlist (IPW-*). Voucher codes are a COULD.
PA-03MUSTPartner offboarding deletes the partner's subscription tuple, which transitively revokes all member access immediately (≤ AC-03 window); membership tuples are also expirable (FGA-02 applies).
PA-04SHOULDPartner usage (reads per partner, unique users) is reported from events for contract management.
PA-05MUSTSeat limits: a partner may carry an optional max_seats; the count of active membership tuples is kept and checked on every membership write — writes beyond the cap are rejected and reported (PA-04). Seat changes are audited (ADM-03).

3.5 Ad-gated authorisation (AG) — grant API + CEP

IDPriorityRequirement
AG-01MUSTThe gate may carry an offer of kind access_grant with method rewarded_ad (decided by the CEP, UP-01; rendering out of scope). Completion of the ad is verified server-side via a signed callback from the ad provider before any access is granted.
AG-02MUSTOn verified completion the grant API (FGA-03) issues a short grant — default: the single triggering article, TTL 24 h. Daily cap per subject (default 2 rewarded unlocks/day) enforced by the grant API.
AG-03SHOULDAd-gated unlocks are logged distinctly (granted_by=ad_gate) so their cannibalization of subscriptions is measurable (AN-14).

3.6 Data-gated authorisation (DG) — grant API + CIAM traits

IDPriorityRequirement
DG-01MUSTThe gate may offer access in exchange for data (newsletter opt-in, survey answer, profile trait) — an access_grant offer with method data_exchange, fulfilled as a progressive-registration step (PR-01). The collected data lives in CIAM traits with an explicit consent + purpose record.
DG-02MUSTOn verified completion the grant API issues the configured grant (default: article or section, TTL 7 days), granted_by=data_gate.
DG-03MUSTGDPR constraint: the exchange must be explicit and specific ("your email for this article") — no bundled consent, and withdrawal of consent stops future use of the data but does not retroactively revoke the consumed grant.

3.7 IP whitelisting (IPW) — edge + FGA

IDPriorityRequirement
IPW-01MUSTPartner IP ranges (CIDR) are configuration (API-03). The edge Worker matches the client IP and forwards a partner_id in the trusted context (INF-01/02) — clients cannot inject it.
IPW-02MUSTThe origin treats a request with partner_id as partner-authorised if the partner's FGA subscription tuple is active (PA-01) — no personal account required. Such reads are metered per partner (PA-04), not per visitor.
IPW-03SHOULDRange changes are audited; overlapping/invalid CIDRs rejected at config load. A range match never reduces access for a visitor who also has a personal entitlement.

3.8 Device limiting (DL) — CIAM sessions + origin enforcement

IDPriorityRequirement
DL-01MUSTA user may have at most max_devices concurrent sessions (config, default 3; per plan tier configurable — e.g. complete 5). A device = one Kratos session with device metadata (user agent, platform, channel).
DL-02MUSTEnforcement at session creation (login): exceeding the limit revokes the oldest session (default) or prompts the user to choose. Revoked sessions die at next token refresh (SM-02), i.e. within the access-token TTL.
DL-03SHOULDConcurrent-streaming-style abuse (one session, many IPs — AC-06) is reported, not auto-blocked, in the experiment phase.

3.9 Session management (SM) — CIAM

IDPriorityRequirement
SM-01MUSTOry Kratos is the session authority; Hydra-issued refresh tokens are bound to a Kratos session. Revoking the session (logout, device limit, admin) invalidates its refresh tokens.
SM-02MUSTRefresh-token rotation with reuse detection: a replayed (stolen) refresh token revokes the whole session chain.
SM-03MUSTThe account page lists active sessions/devices (channel, last seen) with per-session logout and "log out everywhere" (US-07, DL-*).
SM-04MUSTDivision of labor: the edge validates JWTs statelessly (NFR-20) for content access; sensitive operations (AA-01 list) additionally check live session validity against Kratos at the origin — so a revoked session can read for at most the access-token TTL but can never change the account.
SM-05SHOULDAnonymous "sessions" are just the visitor_id cookie (MT-01) — no server-side session object, to keep the anonymous path cheap and cacheable.

4. Visitor identity & metering (MT)

IDPriorityRequirement
MT-01MUSTAnonymous visitors are identified by a first-party cookie containing a random visitor_id (UUID), set on first visit, expiry ≥ 13 months.
MT-02MUSTThe meter state (count + article IDs read this period) is stored server-side, keyed by visitor_id (anonymous) or user_id (logged in). The cookie carries only the identifier.
MT-03MUSTOn login/registration, the anonymous meter state is merged into the user's meter state (take the union of read articles; count = size of union).
MT-04MUSTA read counts toward the meter only when the full article is actually served (not for teaser/gate views, bots, or prefetch requests).
MT-05MUSTKnown crawler user agents (Googlebot etc.) are exempt from metering and gating per the SEO policy (SEO-*), with verification to prevent UA spoofing.
MT-06SHOULDMeter periods and limits are defined per variant configuration (PW-06), defaulting to calendar month / limit 5.
MT-07SHOULDCookie deletion resets the anonymous meter — accepted leak. Track the rate of "new" visitors with old browser fingerprint characteristics only in aggregate, for leak estimation; no covert fingerprinting for enforcement.
MT-08COULDPrivate/incognito detection heuristics may add a soft registration prompt, but must never block free content.
MT-09MUSTCross-channel meter: the meter is per subject, not per channel — reads on web, app, and chat count against the same meter once the subject is identified (API-06).

4.1 Meter management (MT-10..) — origin admin

IDPriorityRequirement
MT-10MUSTMeter parameters (limit, period, per-variant overrides PW-06/MT-06, registered-visitor bonus PW-25) are managed in configuration (API-03) with validation and audit on change.
MT-11MUSTAn internal admin API can inspect a subject's meter state (count, articles, period) and reset it — for support and QA (EX-05). Every reset is audited.
MT-12SHOULDMeter changes (config or reset) take effect within the cache windows (AC-03, INF-05) without deploy or cache purge tooling beyond TTL expiry.

5. Fine-grained authorization (FGA)

Besides subscription entitlements, individual access can be granted by external engines — e.g. the CEP engine rewards an engaged reader with three premium articles, or an AI engine grants access to one recommended article. Grants are stored as relation tuples in the FGA store (Ory Keto, TS-05) and checked in the access decision flow after entitlements.

IDPriorityRequirement
FGA-01MUSTA grant is a relation tuple: subject (visitor or user) — relation (e.g. reader) — object (a specific article, a section/category, or content tier), with metadata: granted_by (source system), reason, expires_at.
FGA-02MUSTEvery grant is time-bound (expires_at required, max TTL configurable, default 30 days) and revocable; expired tuples are cleaned up and no longer match checks.
FGA-03MUSTGrants are written only through a single internal grant API — external engines (CEP, AI) never write to the FGA store directly. The API authenticates the calling system (cf. API-08), validates the grant against policy (allowed relations, object types, TTL bounds, max active grants per subject per source, default 10), and logs an access_granted event.
FGA-04MUSTThe access decision (section 1) calls an FGA check(subject, read, object) after the entitlement check; a positive check serves full content and logs the wall event with outcome fga_grant including granted_by — so CEP- and AI-granted reads are distinguishable in analytics (AN-*).
FGA-05MUSTFGA checks respect the latency budget (API-05): results may be cached per subject+object for ≤ 60 s. If the FGA store is unavailable, the decision proceeds without grants (subscription entitlements unaffected) — degraded, never blocking (cf. NFR-11).
FGA-06MUSTGrants never grant more than read access to content: checkout, account, subscription, and grant-API operations are out of FGA grant scope (no privilege escalation through a compromised CEP/AI integration).
FGA-07SHOULDTypical grant flows: CEP offer of kind access_grant (extends UP-02 kind) accepted → grant API; AI engine recommendation with access → grant API with granted_by=ai_engine and short TTL (e.g. 72 h).
FGA-08SHOULDGrants per subject are visible on the account page ("You have access to…") and listable via an internal admin/debug endpoint with full audit trail (who granted what, when, why).

6. Leak resistance policy & bypass defenses (BP)

The experiment accepts that determined users bypass meters (cookie clearing, incognito). The design goal is honest-majority enforcement: server-side truncation (AC-01) guarantees content is never silently exposed; metering leaks are measured (MT-07), not fought with privacy-invasive techniques. This keeps the experiment GDPR-clean and the results interpretable.

6.2 Bypass defense requirements

IDPriorityRequirement
BP-01MUSTEnforcement is response-level, not rendering-level: no response to an unentitled request — HTML document, API/JSON payload, or asset — contains the premium body; it is absent, not hidden. DOM inspection, view-source, and network-traffic inspection of the Wasm SPA (TS-07) must reveal nothing beyond the server-generated teaser (verifies AC-01/AC-05). The crawler-served HTML rendering (SEO-03) satisfies the same property for unverified requesters.
BP-02MUSTCrawler privileges are granted only on verified identity: Cloudflare verified-bot signal or reverse-DNS + forward-confirm at the edge (INF-01). A spoofed user agent receives the normal gated experience and is flagged in events (AN-05).
BP-03MUSTArchive and scraper bots are excluded from the verified-crawler allowlist. Premium pages send noarchive robots signals. The full text served to verified search crawlers is protected by SEO-01 structured data, distinguishing it from cloaking.
BP-04MUSTRSS/Atom feeds, sitemaps, push notifications, and any syndication endpoint expose at most the teaser for premium content (extends AC-04 to all output channels).
BP-05MUSTThe Referer header is never an input to access decisions. If marketing wants social free passes, they are implemented as signed share tokens: subscriber-generated, article-scoped, expiring URLs that redeem into an FGA grant (FGA-03, granted_by=share_token, capped per subscriber per month).
BP-06SHOULDBypass-rate estimation: a periodic report estimates leakage — e.g. ratio of gated-rendering requests with bypass markers (spoofed UA flags, cookie-reset patterns per MT-07), and spot checks that archive services hold only teasers for sampled premium URLs.
BP-07SHOULDThe decision-matrix test suite (NFR-12) includes bypass regression tests: JS-off rendering, spoofed-UA request, spoofed-Referer request, feed content check, and cache-header check on full premium responses.

7. Gate component behavior

IDPriorityRequirement
AC-10MUSTThe gate renders inline below the teaser; the page remains navigable (header, footer, related free articles).
AC-11MUST"Log in" on the gate opens the auth flow and, on success with a valid entitlement, reloads the full article in place.
AC-12MUST"Subscribe" leads to checkout (PAY-*) with a return URL back to the gated article.
AC-13SHOULDSoft gates (dynamic variant) are dismissible once per session; dismissal is logged as a wall event with outcome dismissed.
AC-14MUSTConsent (cookie) wall step: the gate can be configured to render a consent-first step where legally required (GAP-08). It is a distinct step, never bundled with payment or data-exchange consent (DG-03), and its outcome feeds the consent state of AN-20. Designed in the wall editor (ADM-11).
