Doc 4 — Analytics & Experimentation

The experiment's purpose is to compare paywall strategies. Without solid assignment and measurement, the rest of the system produces no learning.

1. Variant assignment (EX)

IDPriorityRequirement
EX-01MUSTEvery visitor is deterministically assigned to one variant via hash(visitor_id + experiment_id) mod weights — stable across visits, no flicker between variants.
EX-02MUSTAn experiment definition holds: id, name, start/end date, variants (paywall type + configuration, per PW-06), and traffic weights. Stored as configuration, editable without deploy.
EX-03MUSTAssignment survives login: the variant follows the visitor when visitor_id is linked to user_id (US-04). On conflict (user seen with two assignments), the earliest assignment wins.
EX-04MUSTA holdout/control variant (no paywall changes) can be configured to measure baseline behavior.
EX-05SHOULDA debug override (e.g. query param + staff flag, or a cookie set via internal tool) forces a chosen variant for QA, excluded from analytics.

2. Event tracking (AN)

IDPriorityRequirement
AN-01MUSTEvents are logged server-side with: timestamp, visitor_id/user_id, variant, channel (web | app | chat), article_id, event type, and context. No third-party analytics dependency required for core metrics.
AN-02MUSTMinimum event set: page_view, article_read (counted read, MT-04), wall_shown, wall_dismissed, gate_cta_click, register_start/complete, checkout_start/complete, login, cancel, and offer-engine events offer_triggered, offer_suppressed, offer_accepted, offer_declined (see UP-04, DN-06) with the full offer object in context.
AN-03MUSTwall_shown events include the paywall decision context: type, meter state, and (for dynamic) score and threshold (PW-43).
AN-04SHOULDEvents are append-only and exportable as CSV/Parquet for offline analysis.
AN-05SHOULDBot traffic (MT-05) is flagged in events, not silently dropped, so filtering is auditable.

3. Metrics & dashboard (AN)

IDPriorityRequirement
AN-10MUSTA dashboard (internal page) shows per variant: visitors, article reads per visitor, wall events, gate CTR, registrations, checkout starts, conversions, conversion rate (conversions / wall_shown uniques).
AN-11MUSTAlso per variant: reach cost — change in page views and reads vs. control, so conversion gains can be weighed against lost traffic.
AN-12SHOULDDisplay confidence intervals (e.g. Wilson) on conversion rates; flag when sample size is too small to compare variants.
AN-13COULDCohort view: conversion and 30-day retention by week of first visit.
AN-14SHOULDOffer performance view: per offer_id — triggered, accepted, declined, suppressed counts and acceptance rate, broken down by channel; for downsells also retained-revenue estimate vs. plain cancellation.

4. Dynamic paywall scoring (DY)

IDPriorityRequirement
DY-01MUSTPhase 1 heuristic score (0–100) computed from: visits in last 30 days, premium reads this month, days since first visit, referrer type (direct/search/social), and registered y/n. Weights in configuration.
DY-02MUSTScore thresholds map to decisions: < T_soft → open, T_soft..T_hard → soft gate, ≥ T_hard → hard gate. Defaults: 40 / 70.
DY-03MUSTScoring is a pure function of stored engagement data — same input, same score — so decisions are reproducible in analysis.
DY-04SHOULDThe scoring interface allows a trained model to replace the heuristic later (phase 2) without changing callers.
DY-05COULDLog counterfactuals: what each other strategy would have decided on the same page view, enabling offline comparison.
DY-06SHOULDExternal scores as input: the scoring interface (DY-04) accepts externally supplied scores — e.g. a Likelihood-to-Churn score from the CEP/analytics side (GAP-02). The paywall consumes such scores as inputs; it never defines or computes them.

5. Privacy constraints on analytics

IDPriorityRequirement
AN-20MUSTFirst-party, functional + analytics data only; consent banner per GDPR/Telecommunicatiewet where required. No data sale, no third-party ad trackers in the experiment.
AN-21MUSTEvent data is pseudonymous (visitor_id); account deletion (US-07) unlinks or deletes the user's events within 30 days.
