Paywall Experiment — Requirements

Project: paywall (media company experiment) · Version 1.0 · 2026-06-11

This set of documents specifies the functional and technical requirements for a paywall experiment for a media company. The goal of the experiment is to implement and compare four paywall models — hard, metered, freemium, and dynamic — on a single platform, and to measure which converts best.

How to read these documents

Every requirement has a unique ID (e.g. MT-03) and a priority:

PriorityMeaning
MUSTRequired for the experiment to be valid. Build first.
SHOULDImportant, but the experiment can launch without it.
COULDNice to have; only if time permits.
For the coding phase: implement in document order. Doc 2 (access control) is the core engine; Docs 1 and 3 configure and feed it; Doc 4 measures it; Doc 5 constrains how everything is built. Reference requirement IDs in commit messages and code comments.

Scope

In scope: article-level paywall on the web (responsive), four paywall strategies selectable per experiment group, anonymous and logged-in metering, mock or sandbox payment flow, A/B assignment, and a conversion dashboard.

In scope (triggering only): upsell and downsell offer triggering (UP-*/DN-*) — channel-aware (web, app, chat), with offer decisions delegated to an external CEP engine (mocked in the experiment). The presentation of offers (shop-like UI) is a separate feature and out of scope. The experiment implements the web channel; APIs must be channel-agnostic so app and chat plug in later.

Out of paywall scope (owned by adjacent systems): marketing decisioning, journeys, and churn modeling (CEP); coupons, dunning, and billing self-service (subscription management system); credential management (CIAM); offer/shop UI; native apps; print bundles; real payment provider contracts; ad-tech integration; gift subscriptions. Multi-brand is supported for wall rendering and design (Doc 7); the experiment itself runs one brand.

Glossary

TermDefinition
EntitlementA right granted to a user to access content (e.g. "premium", "all-access").
MeterCounter of free premium articles a visitor has consumed in the current period.
Wall eventThe moment the paywall is shown to a visitor instead of (full) content.
GateThe UI component that blocks content and presents the offer.
VariantOne experiment arm: a paywall type plus its configuration.
ConversionA visitor completing checkout and becoming a subscriber.
Propensity scorePredicted likelihood that a visitor will subscribe, used by the dynamic paywall.
Registration wallGate that requires a free account (no payment) to continue.
UpsellOffer to move a subject to a higher-value plan (higher tier, or monthly → annual).
DownsellRetention offer at risk moments (cancel intent, payment failure): lower tier, temporary discount, or pause.
Offer engineLocal component (decide_offer) that queries the CEP, validates the result against guardrails, and returns at most one offer.
CEP engineExternal Customer Engagement Platform that holds the offer rules/campaigns and decides which offer to make for a given trigger; mocked during the experiment.
ChannelThe touchpoint a subject interacts through: web, app, or chat. Carried on every request, offer, and event.
CIAMExternal Customer Identity & Access Management platform (OIDC): the Ory stack — Kratos (identity) + Hydra (token issuance). Owns registration, login, and credentials; issues the JWTs the paywall validates. Self-hosted containers in the experiment.
FGAFine-grained authorization: per-subject, per-object access grants (Ory Keto relation tuples) checked after entitlements — lets external engines grant extra access.
GrantA time-bound FGA tuple giving a subject read access to an article/section/tier, created via the grant API by e.g. the CEP engine or an AI engine.
Edge / WorkerCloudflare Worker that receives every incoming request: routing, channel detection, cookie issuance, JWT validation, caching. Origin = the Kotlin native services behind it.
JWTSigned token issued by the CIAM asserting the user's identity (sub) and claims; validated server-side on every identity-bearing request (NFR-20).
