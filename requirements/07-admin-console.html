Doc 7 — Admin Console & Visual Wall Editor

Scope boundary (GAP-01 decision): this console manages access control and FGA only. It contains no marketing decisioning — offer rules, campaigns, journeys, segmentation, and churn models live in the CEP. The console answers "who may access what, under which mechanical rules, and how does the wall look" — never "what should we offer this user".

1. Admin console — access control & FGA (ADM)

IDPriorityRequirement
ADM-01MUSTThe console is an internal web app over the existing config and admin APIs (API-03, MT-10/11, FGA-03) — it owns no logic of its own; everything it does is also possible via API.
ADM-02MUSTAccess-control configuration management: meter limits and periods (MT-10), paywall-type parameters per variant (PW-06), dynamic thresholds (DY-02), experiment definitions and weights (EX-02) — with validation, diff preview, and audited publish.
ADM-03MUSTFGA management: browse/search grants per subject (FGA-08), create/revoke grants within policy (FGA-03), manage partner objects, membership sources, seat limits (PA-05), and IP allowlists (IPW-01/03) — all writes audited with actor, timestamp, and reason.
ADM-04MUSTSubject inspector for support/QA: given a visitor_id/user_id, show meter state (MT-11), entitlements, active grants, sessions/devices (read-only from CIAM), variant assignment, and recent wall events — with meter reset and grant revoke actions.
ADM-05MUSTConsole access requires staff identity via the same CIAM (Ory) with AAL2 step-up (AA-01) and role-based permissions (viewer / operator / admin). The console is not exposed on public brand domains.
ADM-06SHOULDConfig changes support draft → review → publish with rollback to any previous version (config is versioned per API-03).

2. Visual wall editor — multi-brand (ADM-10..)

Business users compose how the paywall and registration wall look per brand — the gate's appearance is content, not code (GAP-10).

IDPriorityRequirement
ADM-10MUSTMulti-brand: a brand is a first-class entity (theme tokens: colors, logo, fonts; domain; locale). Wall designs are created per brand; every gated rendering resolves brand → wall design → variant overrides.
ADM-11MUSTThe editor visually composes wall designs for paywalls and registration walls from a constrained block library (headline, body copy, price display, CTA buttons, login link, image, legal text, consent block AC-14) — no arbitrary HTML/JS injection; output is a structured wall definition (JSON) rendered by the gate component (AC-10), preserving the no-JS guarantee (BP-01).
ADM-12MUSTLive preview renders the design in all gate contexts: paywall hard/soft, registration wall, consent step, meter-warning banner (PW-23) — per brand, per breakpoint (mobile/desktop), and as anonymous/registered/subscriber.
ADM-13MUSTDraft → publish workflow with versioning, rollback, and audit (who changed which wall of which brand). Publishing invalidates the edge-cached gated renderings of affected pages (INF-03) within 5 minutes.
ADM-14MUSTWall designs are A/B-testable: a variant (EX-02) can reference a specific wall design version, so copy/layout tests run through the standard experiment framework (PW-07).
ADM-15SHOULDPer-brand locale support: designs hold translatable text fields; the rendering picks the brand's locale (experiment: nl-NL, structure ready for more).
ADM-16SHOULDTemplates: a wall design can be saved as a template and instantiated for another brand, keeping brand theme tokens separate from layout/copy.
ADM-17COULDAccessibility lint in the editor: contrast and tap-target checks against WCAG 2.1 AA (NFR-13) before publish.
