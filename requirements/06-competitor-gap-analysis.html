3. Gap backlog — proposed dispositions

IDGapProposalWhen
GAP-01No-code rules/journey builderDecided: overlaps with the CEP. We build an admin UI for the access control and FGA parts only — no marketing decisioning in it (that stays in the CEP). See Doc 7 (ADM-*).Adopt in v1, scoped
GAP-02Churn prediction (Likelihood-to-Churn)Decided: not defined by the paywall. LtC is an external input (from the CEP/analytics side); the paywall can consume it through the scoring interface (DY-06) but does not compute it.External input
GAP-03Coupons / promo codesDecided: belongs to the checkout process and the subscription management system — outside the paywall. The gate only displays whatever offer/price checkout gives it (PW-03, PAY-02 consistency rule).Out of paywall scope
GAP-04GiftingKeep out of v1; design note: a gift = a paid-for FGA grant or prepaid subscription with a redemption token — fits the existing grant API model when prioritized.Backlog
GAP-05Micropayments / pay-later tab & time passesDecided: valid. PW-08 (day pass) promoted to SHOULD: single-purchase passes implemented as purchasable FGA grants (TTL = pass duration). A Supertab-style aggregated tab is a provider integration question — phase 2 variant.Partly v1 / Phase 2
GAP-06Advanced dunningDecided: part of the subscription management system, not the paywall. The paywall only consumes the resulting subscription status (past_due/expired, AC-02) and the payment_failed trigger (DN-03).Out of paywall scope
GAP-07Partner seat limitsDecided: adopt. PA-05 — a limit that is kept and checked: optional max_seats per partner; membership tuple writes beyond the cap are rejected and reported (PA-04).Adopt in v1
GAP-08Cookie/consent wallDecided: valid. Gate variant config: the gate component can render a consent-first step where legally required (AC-14). Kept separate from monetization walls to avoid GDPR "forced consent" risk (DG-03 logic applies).Adopt in v1
GAP-09Self-serve billing historyDecided: part of the self-service domain, not the paywall.Out of paywall scope
GAP-10Visual wall editor, multi-brandNew requirement: a visual editor for paywall/registration wall renderings across multiple brands — templates, copy, theming per brand, preview, draft→publish. See Doc 7 (ADM-10..).Adopt in v1
Scope principle (decided): the paywall owns access control, enforcement, FGA, and wall rendering config; the CEP owns marketing decisioning and churn modeling; the subscription management system owns checkout extras (coupons), dunning, and billing self-service. Adopted into v1: scoped admin UI (GAP-01 → ADM-*), passes (GAP-05), seat limits (GAP-07 → PA-05), consent wall (GAP-08 → AC-14), and the multi-brand visual wall editor (GAP-10 → ADM-10..).
