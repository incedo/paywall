Doc 5 — Technical Architecture

0. Technology stack (TS)

IDPriorityRequirement
TS-01MUSTBackend services are written in Kotlin and compiled to a native binary via GraalVM Native Image (e.g. Ktor or Spring Boot with native support), targeting fast startup (< 100 ms), low memory footprint, and the latency budgets of API-05 / NFR-10.
TS-02MUSTNative-image compatibility is a build constraint from day one: no unregistered runtime reflection/dynamic proxies; libraries chosen must have native-image support; the native binary is built and smoke-tested in CI on every commit — not only at release.
TS-03SHOULDDevelopment may run on the JVM for fast iteration, but the decision-matrix test suite (NFR-12) must also pass against the native binary, since native-image behavior can differ (reflection, resources, time zones).
TS-04MUSTThe CIAM is the Ory stack: Ory Kratos for identity (registration, login, recovery, verification) and Ory Hydra for OAuth2/OIDC token issuance — self-hosted containers in the experiment, swappable for Ory Network in production. All US-* and NFR-20..24 requirements apply to Ory's endpoints (JWKS via Hydra).
TS-05MUSTFine-grained authorization uses Ory Keto (Zanzibar-style relation tuples) as the grant store behind the FGA requirements of Doc 2 (FGA-*), accessed through its check/write APIs behind a local interface.
TS-06SHOULDThe full stack (paywall service, Ory Kratos/Hydra/Keto, database, mock CEP, mailhog) is runnable locally via one docker compose up, plus wrangler dev for the edge Worker (INF-*).
TS-07MUSTThe web channel for human visitors is a Kotlin Multiplatform Compose SPA compiled to Wasm, served as static assets from the edge. The SPA holds no content and makes no access decisions: all article content (teaser or full body) is fetched from origin APIs that enforce AC-01..05 — the premium body is never present in any response to an unentitled request. The admin console (ADM-*) is built from the same Compose source tree. Verified crawlers bypass the SPA entirely (SEO-03).
TS-08MUSTA static no-JS/no-Wasm fallback HTML page is served with the SPA shell (e.g. <noscript> plus a Wasm-support check): visitors whose browser cannot run the SPA see a styled, branded explanation with a support contact — never a blank page. Enforcement is unaffected: the fallback contains no article content (AC-01, BP-01).

0a. Infrastructure — Cloudflare (INF)

Target infrastructure is Cloudflare: all incoming requests (every channel, API-06) enter through Cloudflare Workers at the edge. The Kotlin native services (TS-01) run as the origin behind the Workers — as Cloudflare Containers or another container runtime — since GraalVM native binaries do not run inside Workers themselves (Workers execute JS/WASM).

IDPriorityRequirement
INF-01MUSTA Cloudflare Worker is the single entry point for all requests. It handles: routing, channel detection (API-06), visitor_id cookie issuance (MT-01), verified-crawler detection (SEO-02/MT-05), and JWT validation (NFR-20, JWKS cached at the edge) — then forwards an enriched, trusted context (subject, channel, crawler flag, token claims) to the origin.
INF-02MUSTOrigin services accept requests only from the Worker (mTLS or shared-secret header); identity/channel headers from clients are stripped and re-set at the edge, so the origin can trust the enriched context.
INF-03MUSTCaching policy: the gated/teaser rendering of a premium article is edge-cacheable (it is identical for all unentitled visitors); the full premium rendering is private/no-store at the edge — never cached where an unentitled visitor could receive it (AC-01 extended to the CDN layer). Free articles are publicly cacheable.
INF-04MUSTThe access decision itself stays at the origin (AC-01); the Worker may short-circuit only the safe cases: free articles and crawler traffic. No meter or entitlement logic is duplicated in the Worker.
INF-05SHOULDMeter state (MT-02) remains in the origin database as the source of truth. Workers KV / Durable Objects may be used as a read-through cache of meter/entitlement lookups within the AC-03/FGA-05 freshness windows — an optimization, not a second authority.
INF-06MUSTIf the origin is unreachable, the Worker fails open per NFR-11: serve the edge-cached gated/teaser rendering (never an error page); log the incident. Subscriber full-content requests may show the teaser with a retry hint in this state.
INF-07MUSTSecrets (origin auth, CEP/Ory endpoints) live in Worker secrets / environment bindings and the origin's secret store — never in code or build artifacts.
INF-08SHOULDWorker code is TypeScript, deployed via Wrangler with the same CI rigor as the origin (TS-02): the decision-matrix tests (NFR-12) run against a wrangler dev + origin stack in CI.
INF-09COULDCloudflare bot management signals are forwarded to the origin as additional input for MT-05/AN-05 bot flagging.

1. Architecture principles

IDPriorityRequirement
API-01MUSTThe paywall is a distinct module/service with one entry point: decide(visitor, article) → {access, gate_config, events}. CMS, frontend, and checkout call it; no paywall logic lives in templates.
API-02MUSTStrategies (hard/metered/freemium/dynamic) implement a common interface (strategy pattern) so new types are pluggable.
API-03MUSTAll configuration (PW-06, EX-02, DY-01 weights, plans PAY-01) lives in versioned config (file or table), hot-reloadable, with validation on load.
API-04SHOULDServer-rendered article pages (SSR) so AC-01 truncation and SEO requirements are natural; client-side hydration optional.
API-05MUSTThe decision endpoint responds in < 50 ms p95 (excluding network) so it can sit in the page render path and inside the edge response budget of NFR-15.
API-06MUSTEvery inbound request carries a channel (web | app | chat), propagated through paywall decisions, offer triggering, and events. The experiment implements web; app and chat must work through the same APIs without code changes (channel as data, not branches).
API-07MUSTCEP integration: the CEP engine is reached through a client interface (request_offer(subject, trigger_context) + inbound endpoint for CEP-initiated offers, UP-08). The experiment runs against a mock CEP that serves configurable offers; the real CEP replaces the mock without touching callers (cf. PAY-05).
API-08SHOULDThe inbound CEP endpoint authenticates requests (shared secret or signature, cf. NFR-03) and is idempotent on offer_id + subject_id.

2. Data model (DM)

brand          (id, name, domain, theme_json, locale)                            -- multi-brand (ADM-10)
article        (id, brand_id, title, body, teaser, content_tier, published_at)
wall_design    (id, brand_id, type, version, status, definition_json)            -- wall editor output (ADM-11..13)

assignment     (subject_id, experiment_id, variant, assigned_at)
experiment     (id, name, starts_at, ends_at, variants_json)

grant_meta     (id, subject_id, relation, object, granted_by, reason, expires_at) -- mirrors Keto tuples for audit/TTL cleanup (FGA-*)
IDPriorityRequirement
DM-01MUSTThe model above is the minimum; fields may be added but not removed. event is append-only (AN-04).
DM-02MUSTEntitlements are derived from subscriptions, never set directly — except a staff/comp flag for testing (EX-05).
DM-03SHOULDMeter state and assignments are keyed by an abstract subject_id so anonymous→user merge (MT-03, US-04) is a re-key, not a schema change.

2.1 Event store at scale (DM-04..) — staying event-sourced

Scaling measures change how the log is read, laid out, and derived from — never what is authoritative. The event store remains the single source of truth at every scale.

IDPriorityRequirement
DM-04MUSTEvery cache and read model — including edge KV entries (INF-05) — is a projection: derived, disposable, and rebuildable from events. No scaling measure may introduce a second write authority next to the event store.
DM-05MUSTTag design is the index strategy. Decide-path reads stay bounded regardless of history length: meter events carry a composite subject+period tag (e.g. meter:visitor:abc:2026-06) so a decision model rebuild reads only the current period's events. New hot query paths get new tags at append time — never history scans.
DM-06SHOULDAppend concurrency scales by sharding append serialization per subject (e.g. advisory lock keyed on hash(subject)): appends for different subjects proceed in parallel while the DCB condition check stays serialized exactly where it matters — within one subject.
DM-07SHOULDThe events table is partitioned by month; events of anonymous subjects beyond the cookie horizon (MT-01, 13 months) move to cold storage via the AN-04 export path. The log remains complete and replayable across hot store + archive.
DM-08SHOULDAt scale, projections move from sync in-process to an async subscription (outbox → queue) so the write path never waits on read-model fan-out. The decide path is unaffected — it reads events directly (revisits tech-stack Q-2).
DM-09COULDSnapshots only on evidence: introduce per-model snapshots only when a decision model demonstrably rebuilds from long histories. The period tag (DM-05) is the meter's natural snapshot boundary — do not build snapshotting preemptively.

3. SEO & crawlers (SEO)

IDPriorityRequirement
SEO-01MUSTThe crawler-served rendering (SEO-03) of gated articles includes paywalled-content structured data: JSON-LD with isAccessibleForFree: false and hasPart marking the gated section — so flexible sampling/full serving to Googlebot is not treated as cloaking.
SEO-02MUSTVerified crawlers (reverse-DNS check, not just user agent) receive the full article (MT-05); unverified claimants get the normal gated experience.
SEO-03MUSTDynamic rendering at the edge: human visitors receive the Wasm SPA (TS-07); requests flagged as verified crawlers by the Worker (INF-01) are routed to a server-rendered HTML rendering of the article generated at the origin — full body per SEO-02, with SEO-01 structured data and canonical URL identical to the SPA route. Crawlability is provided by this rendering path, not by the SPA.

4. Security (NFR)

IDPriorityRequirement
NFR-01MUSTNo credential storage in the paywall system — authentication is fully delegated to the CIAM (US-01). On web, tokens are kept server-side or in secure, HttpOnly, SameSite=Lax cookies — never in localStorage. HTTPS only.
NFR-02MUSTNo card data touches our servers — payment fields are provider-hosted (e.g. Stripe Elements), even in sandbox.
NFR-03MUSTWebhook endpoints verify provider signatures (SUB-02) and are idempotent.
NFR-04SHOULDRate limiting on auth and checkout endpoints; CSRF protection on all state-changing forms.

4.1 CIAM & JWT validation (NFR-20..24)

IDPriorityRequirement
NFR-20MUSTEvery JWT is validated on every request that asserts identity: signature against the CIAM's JWKS, iss equals the configured issuer, aud contains this application, exp/nbf honored with ≤ 60 s clock skew, and algorithm restricted to an allowlist (e.g. RS256/ES256; none and HS* rejected).
NFR-21MUSTJWKS is fetched from the CIAM's discovery endpoint and cached (e.g. 1 h) with automatic refresh on unknown kid, so CIAM key rotation requires no deploy.
NFR-22MUSTValidation happens in one shared middleware/library used by all endpoints (pages, APIs, CEP inbound where user-scoped) — no endpoint implements its own token parsing.
NFR-23MUSTIf the CIAM is unreachable: requests with tokens validatable from cached JWKS proceed normally; otherwise the request degrades to anonymous (AC-07). Free content and gated teasers must remain fully available without the CIAM (cf. NFR-11).
NFR-24SHOULDToken validation failures are logged with reason (expired, bad signature, wrong audience…) and rate-limited per source; repeated failures feed the abuse monitoring of AC-06/NFR-04.

5. Non-functional (NFR)

IDPriorityRequirement
NFR-10MUSTGate render adds < 100 ms p95 to article page load; meter/entitlement lookups cached per AC-03.
NFR-11MUSTIf the paywall service fails, articles fail open (content shown, event logged) — never a broken page for readers.
NFR-12MUSTAutomated tests cover the decision matrix: each paywall type × (anonymous, registered, subscriber, expired) × (free, premium) — minimum 32 cases.
NFR-13SHOULDGate component meets WCAG 2.1 AA (focus order, contrast, screen-reader labels).
NFR-14SHOULDStructured logs for every paywall decision with correlation IDs, retained ≥ 90 days for experiment analysis.
NFR-15MUSTCustomer response within 50 ms: every customer-facing request starts its response within 50 ms p95, measured at the edge (request arrival at the Worker → first response byte). SPA assets and gated/teaser payloads are served from edge cache (INF-03); requests that must reach the origin (access decision, full premium content) consume the API-05 decision budget inside this envelope. Measured continuously and alerted on; budget overruns are a release blocker.
NFR-15COULDFeature flag to kill any variant instantly (revert to control) if it misbehaves in production.

6. Suggested build order

  1. Edge Worker skeleton: routing, cookie, channel detection, origin auth (INF-01/02)
  2. Data model + config loading (DM-*, API-03)
  3. Access decision engine with hard + freemium strategies (AC-*, PW-10/30)
  4. Metering + metered strategy (MT-*, PW-20..24)
  5. CIAM integration (Ory Kratos/Hydra) + JWT validation middleware (US-01..03, NFR-20..24, TS-04)
  6. FGA grant store (Ory Keto) + grant API (FGA-*, TS-05)
  7. Accounts + sandbox checkout + entitlements (US-*, PAY-*, SUB-*)
  8. Experiment assignment + event logging (EX-*, AN-01..05)
  9. Dynamic strategy (DY-*, PW-40..43)
  10. Dashboard (AN-10..12), SEO (SEO-*), hardening (NFR-*)
  11. Admin console + visual wall editor (ADM-*)
